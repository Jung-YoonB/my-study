## 데이터베이스 언어 분류

- DQL
    - 데이터를 조회 할 때 사용
    - JOIN, 서브 쿼리 등을 이용하여 더 상세히 조회 가능
    
    ```sql
    SELECT 조회할_컬럼명, 수식(함수), ... 
    FROM 테이블명
    WHERE 조건
    GROUP BY 그룹화
    HAVING 그룹조건
    ORDER BY 기본_오름차순정렬 ASC, 내림차순 DESC;
    
    -- SELECT, FROM 절을 제외하곤 모두 생략 가능
    ```
    
- DML
    - 데이터를 추가, 수정, 제거할 때 사용
    
    ```sql
    -- 데이터 추가
    INSERT INTO 테이블명 (컬럼명1, 2, 3...) VALUES (값1, 2, 3...)
    
    -- 데이터 수정
    UPDATE 테이블명
    SET 수정할_컬럼명 = 수정_값
    WHERE 수정조건
    	--> WHERE 생략 시 지정한 컬럼 데이터 전체 수정
    	
    -- 데이터 삭제
    DELETE FROM 테이블명
    WHERE 삭제조건
    	--> WHERE 생략 시 지정한 테이블 데이터 전체 삭제
    	
    -- 구조적 합치기
    MERGE INTO 대상_테이블명 별명1
    USING (합칠_테이블명 | 서브쿼리) 별명2
    ON 테이블 매핑 조건
    WHEN MATCHED THEN
    	UPDATE SET 별명1.컬럼1 = 별명2.컬럼1, 1.2 - 2.2, ...
    		--> 조건 일치 시 대상 테이블 값 수정
    WHEN NOT MATCHED THEN
    	INSERT (별명1.컬럼1, 1.2, ...) VALUES (별명2.컬럼1, 2.2, ...)
    		--> 조건 불일치 시 대상 테이블에 값 추가
    ```
    
- DDL
    - 데이터 관련 객체를 생성, 수정, 제거할 때 사용
    - 테이블, 유저, 뷰, 시퀀스 등 적용 가능
    
    ```sql
    CREATE 객체 객체명 ~~~
    ALTER 객체 객체명 ~~~
    DROP 객체 객체명 ~~~
    
    -- 테이블 관련 생성, 수정, 제거
    -- 생성
    CREATE TABLE 테이블명 (
    	컬럼명 데이터타입 [DEFAULT 기본값] [제약조건],
    	...
    	테이블_레벨_제약조건,
    	...
    );
    	/*
    		데이터타입 종류
    			- NUMBER
    			- CHAR(고정크기)
    			- VARCHAR2(가변크기)
    			- DATE
    			
    		제약조건 종류
    			- NOT NULL
    			- PRIMARY KEY
    			- FOREIGN KEY
    			- UNIQUE
    			- CHECK
    	*/
    
    -- 컬럼/제약조건 등 추가
    ALTER TABLE 테이블명 ADD (추가내용);
    -- 수정
    ALTER TABLE 테이블명 MODIFY (수정내용);
    --제거
    ALTER TABLE 테이블명 DROP (제거내용);
    -- 이름 변경
    ALTER TABLE 테이블명 RENAME 변경할대상 대상명 TO 변경명;
    
    -- 객체 완전 삭제
    DROP TABLE 테이블명;
    -- 객체는 남겨두고, 데이터 완전 삭제
    TRUNCATE TABLE 테이블명;
    ```

## 정규화 과정 고민

- 다음 상황을 읽어보고, 각자의 생각을 자유롭게 정리해보기
    - **상황**
        - 하나의 ‘주문 테이블’에 [고객 이름, 연락처, 주문 상품, 가격, 배송지]를 
        모두 몰아서 저장하고 있습니다.
    - **생각해볼 부분**:
        - 한 고객이 이사를 가거나 전화번호를 바꾸면 어떤 번거로운 일이 생길까?
            
          ```
            🤔
            
            주문을 많이 한 고객일 수록, 데이터를 일일히 찾아내어 수정해주어야 한다.
            
           ```
            
        
        - 아직 상품을 한 번도 주문하지 않은 신규 가입 회원의 정보는 이 테이블에 저장할 수 있을까?
            
            ```
            🤔
            
            저장 할 순 있으나, 불필요한 NULL 값이 많이 저장 된다.
            
            만약 주문 상품, 가격 등에 NOT NULL 을 적용해 두었다면 저장할 때 
            
            의미 없는 값을 추가로 넣는 것을 고려해야 하므로 데이터가 일관성이 없어진다.
            
           ```
            
        
        - 이 문제를 해결하려면 테이블을 어떻게 나누는 것이 좋을까? 
        (가볍게 글로 설명하거나 그림으로 그려보아도 좋습니다.)
            
            ```
            😊
            
            고객 정보 테이블, 주문 기록 테이블로 나누는 것이 깔끔하다.
            
            고객 정보 테이블에 [고객번호, 고객 이름, 연락처, 배송지] 등을 저장하고
            
            주문 기록 테이블에 [상품번호, 주문 상품, 가격, 주문 고객 번호] 등을 저장한 뒤
            
            두 테이블을 [고객 번호]로 연결해주면  데이터의 일관성을 유지하면서도
            
            정보 변경 시에도 손쉽게 보수성이 좋게 데이터를 관리할 수 있다.
            
            ```
